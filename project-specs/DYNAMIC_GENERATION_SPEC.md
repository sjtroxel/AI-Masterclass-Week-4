# ChronoQuizzr — Dynamic Event Generation Spec

## Overview

This spec replaces the static-pool shuffle in `GET /api/game/start` with live, on-demand event
generation backed by the `ChroniclerEngine`. Every game session receives 5 freshly-generated,
adversarially-verified historical events. The static pool (`events.json` +
`generated_events.json`) is retained exclusively as the **blacklist source** (preventing
duplicates of known events) and as the **fallback** when the Anthropic API is unavailable.

See `project-specs/CHRONICLER_ENGINE_SPEC.md` for the Generate → Adversary → Rewrite loop
that backs this. See `project-specs/ADR_ANTHROPIC_PIVOT.md` for the provider decision.

---

## 1. Motivation

| Problem | Current State | Target State |
|---------|---------------|--------------|
| Repeat events across games | 20-event static pool, Fisher-Yates shuffle — heavy repetition for frequent players | Fresh generation per session — effectively infinite |
| Idle adversarial loop | `ChroniclerEngine` only fires at startup if pool < 5 | Fires on every `GET /start` |
| Two-source event pool | `events.json` + `generated_events.json` both loaded as play sources | Only loaded as blacklist; no longer served directly |

## 2. Architecture

### 2.1 New `GET /api/game/start` Flow

```
Client                  game.ts (route)             eventGenerator.ts / ChroniclerEngine
──────                  ───────────────             ────────────────────────────────────

GET /api/game/start
        │
        ▼
  Build blacklist ────────────────────────────────> All IDs from eventPool (seed + batch + previously generated this session)
        │
        ▼
  Promise.all([
    generateEvent('easy',   blacklist),
    generateEvent('medium',  blacklist),
    generateEvent('hard',    blacklist),
    generateEvent('medium',  blacklist),
    generateEvent('hard',    blacklist),
  ])                       ──────────────────────> ChroniclerEngine.generateVerifiedEvent() × 5 (concurrent)
        │                                           Each call runs full Gen → Adversary → Rewrite loop
        │ ~2–5 seconds
        ◀──────────────────────────────────────── 5 verified HistoricalEvent objects
        │
  Post-generation dedup
  (replace any duplicate IDs with seed fallback)
        │
  Push all 5 into eventPool ◀─── CRITICAL: POST /guess looks up eventPool by id
        │
  Strip hiddenCoords
        │
  res.json(5 × GameEvent)
        │
        ▼
     Client
```

### 2.2 Difficulty Distribution

Each game session receives a balanced difficulty mix:

| Slot | Difficulty |
|------|-----------|
| 1    | `easy`    |
| 2    | `medium`  |
| 3    | `hard`    |
| 4    | `medium`  |
| 5    | `hard`    |

The array is shuffled before returning to the client so difficulty order is not predictable.

### 2.3 Parallel Execution via `Promise.all`

All 5 generation calls are issued concurrently. Since `ChroniclerEngine` has its own internal
retry loop (up to 3 topics × 3 rewrite rounds), each call is self-contained.

**Intra-batch collision risk:** Extremely low — each call independently selects a distinct
historical topic from an arbitrary pool of human history. A post-generation dedup check
(see §2.4) eliminates any residual risk.

```typescript
// Conceptual shape (not final code)
const generated = await Promise.all(DIFFICULTY_SLOTS.map(d => generateEvent(d, blacklist)))
```

### 2.4 Blacklist Construction

The blacklist is built once before `Promise.all` fires:

```
blacklist = eventPool.map(e => ({ id: e.id, locationName: e.locationName }))
```

`eventPool` at this point contains:
- 10 curated seed events (from `events.json`, always loaded)
- 0–10 batch-generated events (from `generated_events.json`, loaded if present)
- Any events generated by previous `GET /start` calls this server session

The `generateEvent()` façade must be updated to accept and forward this blacklist to
`ChroniclerEngine.generateVerifiedEvent(difficulty, alreadyGenerated)` — the engine's
signature already supports this parameter.

### 2.5 Post-Generation Dedup

After `Promise.all` resolves, check for duplicate IDs within the batch of 5:

```
if any two generated events share an id → replace the latter with a random seed event
```

This is a safety net, not an expected path. Log a warning when it fires.

### 2.6 `eventPool` Augmentation — CRITICAL

`POST /api/game/guess` does:

```typescript
const event = eventPool.find((e) => e.id === eventId)
if (!event) { res.status(404)... }
```

Generated events **must be pushed into `eventPool`** before `res.json()` is called. This
guarantees that by the time a client submits a guess (which can only happen after receiving
the `/start` response), the event is findable.

```
After Promise.all resolves:
  eventPool.push(...generated5Events)   // ← adds full HistoricalEvent including hiddenCoords
  res.json(stripped GameEvents)         // ← sends coordinate-free view to client
```

**Memory growth:** Each game adds 5 events (~0.5 KB each). After 10,000 games ≈ 25 MB.
Acceptable for a demo application. A future production spec would add a TTL eviction strategy.

### 2.7 Fallback Strategy

If `Promise.all` rejects (API error, `FatalProviderError`, network timeout, 429):

1. Catch the error and log it (do not surface to client).
2. Fall back silently to the existing Fisher-Yates shuffle of `eventPool`.
3. Return the shuffled selection as-is — client receives events, game continues.
4. Log `[FALLBACK_STATIC_POOL]` to `server/logs/llm_trace.log`.

```
try {
  generated = await Promise.all(...)
  eventPool.push(...generated)
  res.json(strip(generated))
} catch (err) {
  log('[FALLBACK_STATIC_POOL]', err.message)
  const selected = shuffle(eventPool).slice(0, 5)
  res.json(strip(selected))
}
```

**Note:** A `FatalProviderError` (401/403/404) indicates a misconfigured API key and will
affect every subsequent game. The fallback prevents a broken game session, but the server
operator should be alerted. Consider a `console.error` on `FatalProviderError` distinct from
the general catch.

## 5. Breaking Changes to Watch For

### 5.1 `GET /start` is Now Async — Express Handler Must Be `async`

**Before:**
```typescript
router.get('/start', (_req: Request, res: Response) => { ... })
```

**After:**
```typescript
router.get('/start', async (_req: Request, res: Response) => { ... })
```

Express does not automatically catch rejected async handlers in older versions. Wrap the
body in a `try/catch` (see §2.7) — do **not** rely on Express's default error handler
for the fallback path.

### 5.2 `generateEvent()` Signature Change

`eventGenerator.ts` gains a second parameter. All existing callers must be audited:

| Caller | Current call | Action needed |
|--------|-------------|---------------|
| `server/index.ts` | `generateEvent(tiers[i % tiers.length])` | Pass `eventPool` as blacklist (optional param — no change strictly required, but recommended) |
| `server/routes/game.ts` (new) | New call site | Pass full blacklist |

### 5.3 `eventPool` Grows at Runtime

`eventPool` is no longer a fixed-size array. After many game sessions, it will contain
seed events + batch events + all session-generated events. Any code that assumes `eventPool`
is a static snapshot (e.g., hypothetical future admin endpoints, test setup) must account
for this.

### 5.4 Response Latency Increase on `/start`

`GET /start` changes from ~1 ms (array shuffle) to ~2–5 s (LLM calls). Client-side code
that has timeout assumptions (e.g., `AbortController` with a tight timeout) would fail.
Current `GameBoard.tsx` uses the browser's default fetch — no timeout — so this is safe.

### 5.5 `generated_events.json` No Longer Serves Events Directly

After this change, events from `generated_events.json` are never selected for play —
they only appear in the blacklist. The batch script (`npm run generate --prefix server`)
remains useful for warming the blacklist before deploy but does not directly affect
gameplay events.

## 6. Future Considerations (Out of Scope)

- **Per-player seen-event tracking:** Store played event IDs in `localStorage` on the client;
  send them as a `X-Seen-Events` header on `GET /start` so the server can extend the blacklist
  beyond the static pool.
- **`eventPool` TTL eviction:** Remove events older than N hours to bound memory growth.
- **Request timeout:** Add a 30 s `AbortController` timeout to the client's `GET /start` fetch
  with a user-facing "The archives are slow today…" retry prompt.
- **Rate limiting guard:** If Anthropic returns 429, log a warning and extend the static
  fallback window before retrying generation.
